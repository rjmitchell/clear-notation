# CEO Plan: ClearNotation v0.1 — From Spec to Usable Language

Generated by /plan-ceo-review on 2026-04-06

## Current State

The ClearNotation v0.1 spec and reference parser/validator already exist:
- 5 spec documents: handoff, syntax decisions, config contract, examples corpus, AST conformance
- Normative EBNF grammar (`clearnotation-v0.1.ebnf`)
- Reference Python parser/validator in `clearnotation_reference/` (stdlib-only, zero dependencies)
- Manifest-driven test harness in `clearnotation_harness/`: 44/44 fixtures passing
- Reference config (`clearnotation.toml`) and machine-readable built-in registry

All new work extends the existing Python reference implementation. A Rust or Go rewrite is a separate future effort, not part of this plan.

## Vision

### 10x Check

The current v0.1 produces a spec + reference parser. That's a language nobody can use in practice. The 10x version is **a language you can actually write in, today.** Not just a spec that proves correctness. A complete authoring environment where someone can write their first `.cln` file, run a CLI command, and see beautiful output.

The aspirational end-state includes `pip install clearnotation`, `cln init` scaffolding, and a VS Code extension with diagnostics. Those are deferred beyond this plan's scope. This plan delivers the foundation they all depend on.

### Platonic Ideal

A senior engineer in 2027 starts a documentation project. They write a `.cln` file in VS Code with syntax highlighting (tree-sitter), run `cln build`, and get publication-quality HTML. Error messages show the source line, an underline, and a suggestion. Cross-references just work. The CI runs `cln check` and catches every mistake. The language documents itself. The user feels **trust** — nothing silently breaks, the output is consistent, the language is honest about what it supports.

## HTML Renderer Specification

The renderer consumes the normalized AST (from `clearnotation-v0.1-ast-conformance.md`) and produces semantic HTML5. Default element mappings:

| AST Node | HTML Output |
|----------|-------------|
| Document | `<html><head>...<body>` wrapper with meta-derived `<title>` and stylesheet link |
| Heading | `<h1>` through `<h6>` with `id` attribute from slug/anchor |
| Paragraph | `<p>` |
| ThematicBreak | `<hr>` |
| BlockQuote | `<blockquote>` containing `<p>` per quoted line |
| UnorderedList | `<ul><li>` |
| OrderedList | `<ol start="N"><li>` preserving authored start number |
| Text | raw escaped text |
| CodeSpan | `<code>` |
| Strong | `<strong>` |
| Emphasis | `<em>` |
| Link | `<a href="target">label</a>` |
| Note | inline superscript `<sup><a href="#fn-N">[N]</a></sup>`, endnote section at document end with `<li id="fn-N">content <a href="#fnref-N">↩</a></li>` |
| Ref | `<a href="#target">target</a>` (links to heading/anchor ID) |
| Toc | `<nav class="toc">` with nested `<ul><li><a>` reflecting heading hierarchy (all h1-h6 levels included by default in v0.1) |
| Callout | `<aside class="callout callout-{kind}">` with optional `<p class="callout-title">` |
| Figure | `<figure><img src="..."><figcaption>` (all child blocks of Figure rendered inside `<figcaption>`) |
| MathBlock | `<pre class="math"><code>` containing raw math text. Server-side KaTeX rendering is deferred to v0.2. v0.1 preserves the raw text in a styled block. |
| Table | `<table>` with `<thead>/<tbody>`, `<th>/<td>`, `text-align` from `align` attribute |
| SourceBlock | `<pre><code class="language-{lang}">` with HTML-escaped content |
| ExtensionBlock | `<div data-extension="{type}">` with child content |
| ExtensionInline | `<span data-extension="{type}">` |

The HTML output includes a `<!DOCTYPE html>` declaration, `<meta charset="utf-8">`, and a `<link>` to `clearnotation.css`.

## Error Message Specification

Retrofit the existing reference parser in `clearnotation_reference/parser.py` and `clearnotation_reference/validator.py` to carry `(line, column)` positions on all parsed nodes and error reports.

Position data model:
- `file`: source file path (or `<stdin>`)
- `line`: 1-based line number
- `column`: 1-based column (Unicode scalar offset from line start)

Severity levels: `error` (parse/validation failure), `warning` (reserved for future use)

Output format modes:
- **Human** (default, terminal): ANSI-colored, source context, underline carets, suggestion line
- **Plain**: same structure, no ANSI codes
- **JSON**: structured `{"file", "line", "column", "severity", "code", "message", "suggestion"}` for CI tooling

Example (human mode):
```
error[E001]: unknown directive
  --> docs/intro.cln:12:1
   |
12 | ::diagram[kind="flow"]{
   | ^^^^^^^^^^ `diagram` is not registered in clearnotation.toml
   |
   = help: registered block directives: callout, figure, math, table, source
```

Error codes: `E0xx` for parse errors, `E1xx` for validation errors. Each error class has a unique code.

## CLI Specification

The CLI extends the existing Python reference implementation. Entry point: `python -m clearnotation` or `cln` (via setuptools console_scripts).

### `cln build <file|dir>`
- Parses, validates, normalizes, and renders to HTML
- Single file: writes `<basename>.html` next to the source (or `--output <path>`)
- Directory: recursively processes all `.cln` files, writes `.html` next to each source file by default. `--output <dir>` mirrors the source tree into the specified output directory.
- Copies `clearnotation.css` to the output root (same directory as the first output file, or `--output` root). HTML files use a relative `<link>` to the CSS based on their depth.
- Discovers `clearnotation.toml` by walking up from the input file
- Exit codes: 0 = success, 1 = parse/validation error, 2 = I/O error

### `cln check <file|dir>`
- Parses and validates without rendering
- Same discovery and exit codes as `cln build`
- Outputs errors to stderr in the configured format (human/plain/json via `--format`)

### `cln ast <file>`
- Parses, validates, normalizes, and outputs the normalized AST as JSON to stdout
- JSON schema follows the normalized AST shape from `clearnotation-v0.1-ast-conformance.md`
- Exit codes: 0 = success, 1 = parse/validation error

### Common flags
- `--config <path>`: explicit config file path (overrides discovery)
- `--format human|plain|json`: error output format (default: human if TTY, plain otherwise)
- `--help`, `--version`

## Default Stylesheet Specification

`clearnotation.css` is a single CSS file that makes unstyled HTML output look publication-quality.

Goals:
- System font stack (no web font dependencies)
- Max content width: 48rem centered
- Typography: 1.125rem base, 1.6 line-height, modular scale for headings
- Light theme by default, `prefers-color-scheme: dark` media query for dark mode
- Code blocks: monospace, subtle background, horizontal scroll on overflow
- Callouts: colored left border by kind (info=blue, warning=amber, danger=red, tip=green)
- Tables: subtle borders, alternating row backgrounds, header bold
- Notes: superscript numbers, endnote section at the bottom with horizontal rule separator
- Figures: centered, caption below in smaller italic text
- Math blocks: monospace, centered, subtle background (until KaTeX integration in v0.2)
- TOC: indented list, no bullets, subtle styling
- Responsive: readable on mobile without a separate layout (content reflows naturally at 48rem max)

No build step. No preprocessor. One `.css` file that works with a `<link>` tag.

## Tree-sitter Grammar: Parameterized Registry Challenge

ClearNotation's EBNF is parameterized by the directive registry (directive names, placement, body mode are resolved from `clearnotation.toml`). Tree-sitter grammars compile to static C parsers and cannot be parameterized at runtime.

Resolution: The tree-sitter grammar highlights the **syntactic frame** without knowing the registry:
- `::name[attrs]{body}` is highlighted as a directive regardless of whether `name` is registered
- `::name[attrs]` (no body) is highlighted as a self-closing directive
- Directive names, attribute keys/values, body content all get distinct highlight groups
- The tree-sitter grammar does NOT validate directive names, body modes, or attribute schemas

Semantic validation (unknown directives, body mode mismatches, attribute errors) is handled by a future LSP server, not the tree-sitter grammar. The tree-sitter layer provides syntax highlighting and structural folding only.

This means tree-sitter will parse any `::name` as a valid directive frame. Users get immediate visual feedback (coloring) but semantic errors only appear when running `cln check` or (eventually) an LSP. This is the same split VS Code uses for TypeScript (tree-sitter for highlighting, tsserver for diagnostics).

## Acceptance Criteria

### Migration cheat sheet
A single `.cln` file covering the 15 most common Markdown constructs and their ClearNotation equivalents. Rendered by the v0.1 pipeline to HTML. Includes: headings, bold, italic, code, links, images (via figure), lists, blockquotes, code fences, horizontal rules, tables, footnotes (via notes), and callouts (via directives). Each entry shows the Markdown syntax and the ClearNotation equivalent side-by-side.

### Self-documenting spec
Convert `clearnotation-v0.1-syntax.md` and `clearnotation-v0.1-examples.md` to `.cln` format. Render them to HTML using the v0.1 pipeline. The rendered output must be readable and complete (all spec content preserved, using only v0.1 features). This validates the renderer against real, complex documents.

### Tree-sitter grammar
Grammar in `tree-sitter-clearnotation/`. Ships with `grammar.js` and `highlights.scm` queries file. Acceptance: `tree-sitter parse` succeeds on every valid and validate-invalid fixture file (parse-invalid fixtures are expected to produce tree-sitter ERROR nodes, since they contain intentionally broken syntax). The grammar highlights directive frames, inline constructs, attributes, and code fences with distinct highlight groups.

## Implementation Sequence

0. **Fix error types dependency inversion** — move `ParseFailure` and related error types from `clearnotation_harness` to `clearnotation_reference.errors`. Update the harness to import from the reference package. This unblocks everything else.
1. **Normalized AST model** — add typed normalized node classes to `clearnotation_reference/models.py`: `Callout`, `Figure`, `MathBlock`, `Table` (with `TableRow`, `TableCell`), `SourceBlock`, `Toc`, `Ref`, `ExtensionBlock`, `ExtensionInline`, plus the wrapper `Document` with `MetaMap`. These types don't exist yet. The parsed tree uses generic `BlockDirective`; the normalizer must convert these into typed nodes.
2. **Normalizer** — new module `clearnotation_reference/normalizer.py`. Converts validated parsed tree → normalized AST. Creates typed nodes from generic `BlockDirective` instances. Consumes anchors into block IDs, resolves includes (recursive parse with cycle detection via visited-path set), assigns note numbers, generates heading slugs. Extracts normalization-like logic that currently lives in `validator.py` (heading IDs, note numbering, anchor consumption). Table cell inline parsing moves from validator to normalizer (cells are split from raw text, then parsed with core inline grammar to produce `TableCell` with `Inline[]`). The validator is refactored to do pure validation (reject invalid states) without mutation.
3. **Rich error messages** — retrofit `(line, column)` positions into parser and normalizer. Line-level positions: trivial (block parser already indexes by line). Column-level positions: requires threading original line offset through `InlineParser` child instantiations (4 call sites that slice `self.text[self.pos:]` and lose offset information). Phased: line-level first, column-level refined as error tests are written. Add error formatting module (`clearnotation_reference/diagnostics.py`) with human/plain/JSON output modes.
4. **HTML renderer** — new module `clearnotation_reference/renderer.py`, consumes normalized AST, produces semantic HTML5 per the mapping above.
5. **Default stylesheet** — `clearnotation.css`, ships alongside rendered HTML.
6. **CLI + minimal packaging** — `clearnotation_reference/cli.py` with `cln build`, `cln check`, `cln ast` commands. Add `pyproject.toml` with console_scripts entry point so `cln` works after `pip install -e .` (local dev install). PyPI publishing remains deferred.
7. **Migration cheat sheet** — written in `.cln`, rendered by the pipeline (validation gate).
8. **Self-documenting spec** — convert syntax.md and examples.md to `.cln`, render to HTML (dogfooding gate). If conversion surfaces parser bugs, fix them. If it surfaces spec gaps (features v0.1 doesn't support), document them but don't implement — the converted `.cln` files use only v0.1 features, with some content simplified where necessary.
9. **Tree-sitter grammar** — `tree-sitter-clearnotation/` directory, based on the EBNF, highlights syntactic frame only. Parallel track with steps 7-8. **Does not block the core deliverable.** If tree-sitter slips, the CLI + renderer + stylesheet still ships.

## Landscape Context

- Markdown: ubiquitous but architecturally broken (no formal grammar, ambiguous parse, flavor fragmentation)
- AsciiDoc: powerful but complex, steep learning curve
- Djot: MacFarlane's post-CommonMark experiment, similar goals but less rigorous (no normative EBNF)
- MyST: extends Markdown rather than replacing it
- ClearNotation's differentiator: normative EBNF, fail-closed parsing, typed extensibility, clean-sheet design

## Risk

The #1 risk is adoption, not engineering. The spec is well-designed. The reference implementation works. But every markup language competes with Markdown's network effects. The accepted expansions (error quality, stylesheet, migration guide, tree-sitter, self-documenting spec) are all directly aimed at reducing the adoption barrier.

The #2 risk is tree-sitter scope creep. The grammar is L-sized and the parameterized registry creates a real design challenge. The resolution (syntactic frame only, semantic validation deferred to LSP) is practical but may feel incomplete to users who expect editor errors.

## Reviewer Concerns

These items were raised during adversarial spec review and are noted for implementation:

- **Error code catalog:** The plan defines E0xx/E1xx numbering but doesn't assign specific codes to each error class. Assignment is an implementation detail. The first implementation should produce a skeleton catalog as it implements each error path.
- **Callout child block rendering:** Callout has `blocks: Block[]`. All child blocks are rendered recursively inside the `<aside>`. This is implicit in the AST-to-HTML mapping but worth noting for the implementer.
- **Column-level position tracking:** Retrofitting column positions into the inline parser may require restructuring (the inline parser currently operates on stripped strings). The phased approach (line-level first) mitigates this risk.

## Eng Review Decisions (from /plan-eng-review)

These architectural decisions were made during the eng review and should be followed during implementation:

1. **Two-pass validator/normalizer design:** The validator walks the tree read-only (attribute validation, ref collection, structural checks). The normalizer walks it again for mutations (ID assignment, note numbering, typed node creation). Clean phase separation.
2. **Table cells parsed twice:** The validator parses table cells from raw text for ref collection. The normalizer parses them again to create typed `TableCell` nodes. Double parsing is cheap and avoids shared mutable state between phases.
3. **Hybrid testing:** Golden HTML files in the fixture system (harness). Unit tests for normalizer, diagnostics, and CLI internals in `tests/`.
4. **I/O error handling at CLI layer:** `IOError`, `UnicodeDecodeError`, and `PermissionError` are caught at the CLI entry point and presented as clean error messages (exit code 2). No raw Python tracebacks.
