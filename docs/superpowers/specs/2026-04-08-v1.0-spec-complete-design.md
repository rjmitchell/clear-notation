# ClearNotation v1.0: Spec-Complete Design

**Date:** 2026-04-08
**Goal:** Freeze the ClearNotation language specification at v1.0. All deferred v0.1 features are resolved. The EBNF is normative. Other implementations can build against the spec with confidence it won't change.
**Approach:** Feature-by-feature cycles (spec-update, implement, test, ship as PR). Spec freezes when the last feature lands.

## Scope

**In v1.0:**
1. Inline comments (`// comment` at end of line)
2. Include inlining (`::include` resolves and inlines target content)
3. Nested lists (indentation-based sub-items)
4. Multi-paragraph list items (blank line + indented continuation)
5. Tree-sitter grammar bug fixes (colon-in-prose, `}:` pattern)
6. Spec freeze with new v1.0 documents
7. Conformance suite expansion (70+ fixtures)

**NOT in v1.0:**
- Nested blockquotes (`> > double-nested`) -- adds complexity without sufficient value for technical docs
- Definition lists -- not part of the CLN design philosophy; use tables or callouts instead
- JS/Python rendering parity -- Python is the reference; JS is best-effort
- Tree-sitter grammar expansion for new features -- editor stays on v0.1 grammar, bug fixes only
- Visual editor directive rendering -- post-v1.0 tooling polish

## Implementation Order

1. Inline comments (smallest, block-level precedent exists)
2. Include inlining (well-scoped, validation already works)
3. Nested lists + multi-paragraph items (hardest, most parser changes)
4. Tree-sitter bug fixes (mechanical)
5. Spec freeze + conformance expansion + v1.0.0 tag

## Feature 1: Inline Comments

Extend the existing `//` block-level comment syntax to end-of-line position.

### Syntax

```
# Heading  // this is an inline comment
- List item  // comment on list item
Some paragraph text.  // trailing comment
```

### Rules

- `//` preceded by at least one space or tab starts an inline comment
- Everything from `//` to end of line is the comment
- `//` inside code spans, raw directive bodies, or attribute strings is literal
- `//` at the start of a line (after optional whitespace) remains a block comment
- Inline comments are stripped during normalization, same as block comments
- The parser stores the comment text in the existing Comment node, associated with the line
- The formatter preserves inline comments in their original position

### EBNF Change

The `line_end` production gains an optional inline comment before NL:

```
line_end = opt_ws, [ inline_comment ], NL ;
inline_comment = "//", { comment_char } ;
```

## Feature 2: Include Inlining

`::include[src="path.cln"]` currently validates path existence but does not inline content. For v1.0, the normalizer resolves includes by recursively parsing and normalizing the target file.

### Behavior

- The normalizer reads the target file, parses it with the same registry, validates, normalizes
- The resulting blocks replace the `::include` node in the normalized AST
- The target file's `::meta` block is discarded (only root document's meta matters)
- Circular includes detected and produce a validation error
- Include depth capped at 10 levels (validation error if exceeded)
- Relative paths resolve from the including file's directory, constrained to project root (existing)

### AST Change

`::include` no longer produces `None` in normalized output. It produces the inlined blocks directly, as if written in the parent document.

### Heading ID Collision

Inlined content may introduce duplicate heading slugs. The existing slug collision algorithm (`-2`, `-3`, etc.) applies across the merged document.

### Error Handling

- Missing file: validation error (existing, unchanged)
- Circular include: new error kind `circular_include`
- Depth exceeded: new error kind `include_depth_exceeded`
- Parse/validation error in target: diagnostic includes target file path

## Feature 3: Nested Lists and Multi-Paragraph List Items

The largest change. The v0.1 parser is line-oriented (one line = one list item, no nesting). v1.0 adds indentation-sensitive list parsing.

### Syntax

```
- Top-level item
  - Nested item (2-space indent)
    - Doubly nested (4-space indent)
  - Back to first nesting level

1. Ordered top-level
   - Mixed: unordered inside ordered (3-space indent, aligns after "1. ")

- Multi-paragraph item
  starts here.

  This is the second paragraph of the same item.
  Continuation lines indented to the item's content column.

  - And this is a nested sub-item.

- Next top-level item (back to zero indent)
```

### Rules

- Indent unit: 2 spaces per nesting level for unordered (`- `), content-column alignment for ordered (`1. ` = 3 spaces)
- Tabs are NOT valid indentation for nesting (avoids tab-width ambiguity)
- A blank line followed by an indented line continues the current item as a new paragraph
- A blank line followed by a non-indented or less-indented line ends the list
- Nested items can contain: paragraphs, sub-lists, blockquotes, code blocks, block directives
- Maximum nesting depth: 6 levels (validation error if exceeded)
- Ordered list numbering is per-level (nested ordered lists restart from authored numbers)

### Parser Changes

- `_parse_unordered_list` and `_parse_ordered_list` become recursive, tracking indent level
- New `_parse_list_item_body` collects continuation lines and nested blocks at current indent
- Flat `UnorderedList.items: list[list[InlineNode]]` changes to `UnorderedList.items: list[ListItem]`

### AST Model Changes

- New `ListItem` parsed node: `children: list[InlineNode]` (first-line content) + `blocks: list[BlockNode]` (continuation paragraphs, nested lists, etc.)
- New `NListItem` normalized node mirrors this structure
- `NUnorderedList.items` and `NOrderedList.items` become `list[NListItem]`

### Renderer Changes

- Nested lists produce nested `<ul>`/`<ol>` in HTML
- Multi-paragraph items produce multiple `<p>` inside `<li>`

### Migration

This is a backwards-compatible change. Flat v0.1 lists (no indentation, no blank lines) parse identically under the new rules. `ListItem` with empty `blocks` is equivalent to the old flat item.

## Feature 4: Tree-Sitter Bug Fixes

Scoped to fixing known parse failures. No grammar expansion.

### Known Bugs

- `}:` pattern: grammar misparses close-brace + colon as directive-related token
- Audit `text` rule for other over-restricted characters

### Approach

- Fix the `text` rule to handle `}` followed by `:` correctly
- Rebuild WASM
- Run all valid CLN fixtures through `tree-sitter parse`, verify zero ERROR nodes

### Not in Scope

- Editor visual rendering of directives (post-v1.0)
- Tree-sitter grammar for nested lists or inline comments (post-v1.0)
- New tree-sitter corpus tests for v1.0 features

## Feature 5: Spec Freeze and Conformance Expansion

After all features are implemented and tested.

### New Spec Documents

- `clearnotation-v1.0.ebnf` -- normative grammar (copy from v0.1, add new productions)
- `clearnotation-v1.0-syntax.md` -- all decisions documented, no "deferred" items, explicit "NOT in v1.0" section
- `clearnotation-v1.0-ast-conformance.md` -- updated AST model with ListItem, NListItem, inline comment stripping, include inlining

v0.1 spec files preserved as-is (shipped artifact for existing implementations).

### Conformance Expansion

New fixtures:
- Nested lists (2-3 levels, mixed ordered/unordered)
- Multi-paragraph list items
- Include inlining (single-level, nested includes, circular detection error case)
- Inline comments (in paragraphs, headings, list items, inside directives)
- All with .cln + .ast.json + .html triplets

Target: 70+ total conformance fixtures (up from 54).

### Success Gate

- VERSION reads 1.0.0
- All conformance fixtures pass
- All Python tests green
- Spec documents have no unresolved "deferred" items

## JS/Python Parity

Python is the normative reference implementation. The JS renderer (clearnotation-js) is best-effort for v1.0. The 13 known conformance gaps are tracked in `clearnotation-js/src/conformance.test.ts` as known gaps. Fix incrementally, don't gate release.

## Architecture Impact

### Parser changes (Feature 3 is the risk)

Nested lists transform the parser from line-oriented to indent-aware for list contexts only. The rest of the parser (headings, paragraphs, directives, blockquotes) stays line-oriented. This is a contained change but it touches the core parsing loop.

### Model changes

`ListItem` / `NListItem` are new types added to `BlockNode` / `NormalizedBlock`. All consumers that iterate blocks need to handle the new types (normalizer, validator, renderer, formatter, indexer). The explicit Comment skip pattern established in v0.9.2 is the template for how to add these.

### Backwards Compatibility

All changes are additive. Valid v0.1 documents are valid v1.0 documents. The v1.0 parser accepts a strict superset of v0.1 syntax.
