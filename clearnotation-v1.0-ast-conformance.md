# ClearNotation v1.0 AST And Conformance

This document defines the working v1.0 AST model and the conformance boundary for ClearNotation processors. It complements the grammar in [clearnotation-v1.0.ebnf](/Users/ryan/projects/clear-notation/clearnotation-v1.0.ebnf), the syntax decisions in [clearnotation-v1.0-syntax.md](/Users/ryan/projects/clear-notation/clearnotation-v1.0-syntax.md), the config contract in [clearnotation-v0.1-config.md](/Users/ryan/projects/clear-notation/clearnotation-v0.1-config.md), and the example corpus in [clearnotation-v0.1-examples.md](/Users/ryan/projects/clear-notation/clearnotation-v0.1-examples.md).

## Model overview

ClearNotation has two tree layers in v1.0:

- a parsed document tree, produced by syntax alone
- a normalized document AST, produced after validation and trusted transforms

Renderers consume the normalized AST, not raw source text and not unresolved directives.

## Phase outputs

### 1. Parse output

The parse phase produces a syntax tree that preserves author intent without performing semantic resolution.

It must preserve:

- block order
- inline order
- directive names
- raw attribute values
- raw directive bodies
- source-authored ordered-list numbers
- the presence and position of notes, refs, and anchors

It must not decide:

- whether an attribute name is allowed
- whether an attribute type matches a directive schema
- whether a ref target resolves
- whether an include path is permitted
- whether a generated ID collides

### 2. Validated normalized AST

The validate and transform phases convert the parsed tree into a normalized AST that is safe for rendering.

After normalization:

- `::meta` has become structured document metadata
- `::anchor` has been consumed into block IDs
- `::include` has been recursively resolved — target content is parsed, validated, normalized, and inlined; target meta is discarded
- circular includes have been detected and rejected as normalization errors
- include depth exceeding 10 levels has been detected and rejected as a normalization error
- heading slugs have been deduplicated across the merged document (including inlined content)
- note numbering has been assigned and continues across includes
- built-in directives have become typed core nodes
- unknown or disallowed directives have already failed
- no unresolved raw HTML markup exists as an executable rendering surface

## Parsed tree shape

The parsed tree can be represented with these conceptual node families:

```text
ParsedDocument
  meta: ParsedMetaBlock?
  blocks: ParsedBlock[]

ParsedBlock
  = ParsedHeading
  | ParsedParagraph
  | ParsedThematicBreak
  | ParsedBlockQuote
  | ParsedUnorderedList
  | ParsedOrderedList
  | ParsedSourceFence
  | ParsedBlockDirective
  | ParsedComment

ParsedUnorderedList
  items: ParsedListItem[]

ParsedOrderedList
  items: ParsedListItem[]

ParsedListItem
  content: ParsedInline[]
  blocks: ParsedBlock[]

ParsedInline
  = ParsedText
  | ParsedCodeSpan
  | ParsedStrong
  | ParsedEmphasis
  | ParsedLink
  | ParsedNote
  | ParsedInlineDirective
```

This layer is conceptual rather than wire-format normative. Implementations may represent it differently internally, but they must preserve the same parse-visible distinctions.

## Normalized AST shape

The normalized AST is the renderer-facing model for v1.0.

```text
Document
  meta: MetaMap
  blocks: Block[]

Block
  = Heading
  | Paragraph
  | ThematicBreak
  | BlockQuote
  | UnorderedList
  | OrderedList
  | Toc
  | Callout
  | Figure
  | MathBlock
  | Table
  | SourceBlock
  | ExtensionBlock

Inline
  = Text
  | CodeSpan
  | Strong
  | Emphasis
  | Link
  | Note
  | Ref
  | ExtensionInline
```

## Core node definitions

### Document

```text
Document {
  type: "document"
  meta: MetaMap
  blocks: Block[]
}
```

`MetaMap` is a flat key/value map using dotted keys as literal keys, for example `page.lang`.

### Heading

```text
Heading {
  type: "heading"
  level: 1..6
  id: string
  content: Inline[]
}
```

Rules:

- every heading has an `id` after validation
- if no explicit `::anchor` applies, `id` is generated by the slug algorithm
- if the generated slug is empty and no explicit anchor exists, validation fails

### Paragraph

```text
Paragraph {
  type: "paragraph"
  content: Inline[]
}
```

Rules:

- contiguous paragraph lines normalize into a single inline sequence
- line boundaries inside a paragraph normalize to a single ASCII space

### ThematicBreak

```text
ThematicBreak {
  type: "thematic_break"
}
```

### BlockQuote

```text
BlockQuote {
  type: "blockquote"
  lines: Inline[][]
  id?: string
}
```

Rules:

- a flat blockquote preserves authored line boundaries
- each quoted line is one inline sequence

### UnorderedList and OrderedList

```text
UnorderedList {
  type: "unordered_list"
  items: ListItem[]
  id?: string
}

OrderedList {
  type: "ordered_list"
  items: OrderedListItem[]
  id?: string
}

ListItem {
  content: Inline[]
  blocks: Block[]
}

OrderedListItem {
  ordinal: integer
  content: Inline[]
  blocks: Block[]
}
```

Rules:

- list items may contain nested blocks (sub-lists, continuation paragraphs)
- nesting uses 2-space indent for unordered, marker-width for ordered
- tabs in list indentation are parse errors
- ordered list ordinals preserve the literal authored numbers

### Toc

```text
Toc {
  type: "toc"
  id?: string
}
```

### Callout

```text
Callout {
  type: "callout"
  kind: string
  title?: string
  compact?: boolean
  blocks: Block[]
  id?: string
}
```

### Figure

```text
Figure {
  type: "figure"
  src: string
  blocks: Block[]
  id?: string
}
```

Rules:

- `src` is required
- the parsed body becomes figure child blocks, typically caption content

### MathBlock

```text
MathBlock {
  type: "math"
  text: string
  id?: string
}
```

Rules:

- `text` preserves raw body text with normalized line endings
- the closing line is not part of `text`

### Table

```text
Table {
  type: "table"
  header: boolean
  align?: ("left" | "center" | "right")[]
  rows: TableRow[]
  id?: string
}

TableRow {
  cells: TableCell[]
}

TableCell {
  content: Inline[]
}
```

Rules:

- each source row must normalize to the same number of cells
- when `header = true`, the first row is the header row
- each cell is split from raw text first and then parsed with the core inline grammar

### SourceBlock

```text
SourceBlock {
  type: "source"
  language: string
  text: string
  id?: string
}
```

Rules:

- fenced code blocks and any built-in `source` directive normalize to the same node shape
- `language` is required
- `text` preserves raw body text with normalized line endings

### Inline nodes

```text
Text {
  type: "text"
  value: string
}

CodeSpan {
  type: "code"
  value: string
}

Strong {
  type: "strong"
  content: (Text | CodeSpan)[]
}

Emphasis {
  type: "emphasis"
  content: (Text | CodeSpan)[]
}

Link {
  type: "link"
  label: Inline[]
  target: string
}

Note {
  type: "note"
  number: integer
  content: Inline[]
}

Ref {
  type: "ref"
  target: string
}
```

Rules:

- strong and emphasis remain restricted-content nodes in the normalized AST
- note numbers are assigned by first appearance in depth-first document order
- refs preserve their canonical target string after validation

## Addressable blocks

`::anchor` attaches to the next addressable block. In v1.0, addressable blocks are:

- heading
- paragraph
- blockquote
- unordered list
- ordered list
- toc
- callout
- figure
- math
- table
- source

If an `::anchor` is not followed by one of those blocks, validation fails.

## Built-in normalization rules

The core built-ins normalize as follows:

- `::toc` becomes `Toc`
- `::ref[target="..."]` becomes `Ref`
- `::anchor[id="..."]` is consumed and sets the next addressable block `id`
- `::include[src="..."]` is recursively parsed, validated, and normalized; the resulting blocks replace the include node; the target's `::meta` is discarded; circular includes and depth > 10 are errors
- `::callout[...]` becomes `Callout`
- `::figure[...]` becomes `Figure`
- `::math{...}` becomes `MathBlock`
- `::table{...}` becomes `Table`
- fenced code blocks become `SourceBlock`

## Extension nodes

Extensions may emit only approved typed nodes, never renderer-native markup.

For v1.0, that means an implementation may support:

- `ExtensionBlock { type: string, data: map, blocks?: Block[], inlines?: Inline[], id?: string }`
- `ExtensionInline { type: string, data: map, content?: Inline[] }`

Rules:

- extension node types must come from trusted, registered extensions
- extension nodes must still be serializable as typed AST data
- renderers may reject unknown extension node types unless the project explicitly enables them

## Conformance requirements

### Parser conformance

A parser is conforming for v1.0 if it:

- accepts every `valid` and `validate-invalid` case in [clearnotation-v0.1-examples.md](/Users/ryan/projects/clear-notation/clearnotation-v0.1-examples.md)
- rejects every `parse-invalid` case in that same corpus
- preserves directive placement and body-mode distinctions from the trusted registry
- treats raw HTML source as plain text, not executable syntax

### Validator conformance

A validator is conforming for v1.0 if it:

- rejects every `validate-invalid` case in the example corpus
- enforces unknown-attribute failure
- enforces attribute type checks
- enforces trusted-registry availability
- enforces include path rules
- generates deterministic slugs
- assigns stable note numbers by source order
- resolves refs or fails closed

### Normalizer conformance

A normalizer is conforming for v1.0 if it:

- produces only core or approved extension node types
- consumes anchors and includes before render
- normalizes fenced code and built-in source forms to the same `SourceBlock` shape
- normalizes table cells through the core inline grammar
- resolves includes recursively with circular detection and depth cap
- deduplicates heading slugs across the merged document
- continues note numbering across includes

### HTML renderer conformance

An HTML renderer is conforming for v1.0 if it:

- consumes the normalized AST
- escapes literal source HTML
- preserves heading levels and IDs
- preserves list order and authored ordered-list ordinals
- preserves link targets and ref targets
- preserves note numbering and note order
- preserves source language tags
- preserves table header semantics and cell order

Exact HTML element choice is partially implementation-defined as long as the document semantics are preserved.

### PDF renderer conformance

PDF support is secondary in v1.0.

A PDF renderer is conforming if it preserves the same normalized document semantics as the HTML renderer, even if pagination, line wrapping, and page-level layout differ between implementations.

## Determinism requirements

For the same source document, trusted registry, include tree, and render target:

- parse output must be deterministic
- validation outcome must be deterministic
- slug generation must be deterministic
- note numbering must be deterministic
- normalized AST output must be deterministic

This is a core part of the fail-closed design.

## Out of scope for v1.0 conformance

These are intentionally left outside the first conformance target:

- byte-for-byte HTML equality across implementations
- page-identical PDF output across implementations
- source map formats
- diagnostic message text
- editor behaviors

That keeps conformance focused on syntax, validation, normalized semantics, and safe extensibility.
